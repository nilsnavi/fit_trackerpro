# Playbook: откат БД и приложения после неудачного деплоя

Alembic: каталог `database/migrations/versions/`. Продакшен обычно на `docker-compose.prod.yml` с тегом образов **`IMAGE_TAG`** в `.env` на хосте.

Связанные документы: `docs/db/migration-safety.md`, `docs/db/schema-governance.md`.

---

## Когда откатывать

Принимайте решение об откате, если выполняется хотя бы одно из условий ниже (и нет быстрого безопасного hotfix).

| Критерий | Что смотреть |
|----------|----------------|
| **Ошибка healthcheck** | Контейнер `backend` не переходит в `healthy`, Docker/Kubernetes перезапускает сервис по кругу. |
| **Ошибка readiness / liveness** | `GET /api/v1/system/ready` стабильно **503** или таймауты; при этом до релиза был стабильный **200**. |
| **Ошибка миграции** | Job миграции (например, GitHub Actions «Database Migration») завершился с ненулевым кодом; в логах traceback Alembic при `upgrade`/`downgrade`. |
| **Деградация метрик** | Резкий рост 5xx, ошибок БД, latency после выката; корреляция по времени с миграцией/деплоем; SLO под угрозой при отсутствии объяснения инфраструктурой. |

Перед откатом: зафиксируйте текущую ревизию (`alembic current`), версию образа и путь к последнему pre-migrate дампу (если деплой/миграция его создавали).

---

## Быстрый откат (цель: до 5 минут)

Подходит, если **downgrade** для последней миграции реализован и совместим с предыдущей версией кода, данные не теряются при откате DDL.

1. На хосте деплоя, в каталоге репозитория (например `~/fittracker-pro`):

   ```bash
   docker compose -f docker-compose.prod.yml exec -T backend alembic downgrade -1
   ```

2. В `.env` верните **`IMAGE_TAG`** на предыдущий стабильный тег и подтяните образы:

   ```bash
   docker compose -f docker-compose.prod.yml pull
   docker compose -f docker-compose.prod.yml up -d
   ```

3. Убедитесь, что ревизия БД и код согласованы: `alembic current` внутри `backend` должен соответствовать ожидаемой ревизии для этого тега.

Если откат схемы на шаг назад невозможен или рискован — переходите к разделу про дамп и восстановление.

## Откат с потерей данных (destructive migration)

Используйте, когда миграция **необратима** (DROP COLUMN/TABLE, сжатие типов с потерей значений и т.п.) или `downgrade` неприменим.

### Перед деплоем (политика)

- Всегда делайте **`pg_dump`** (или эквивалентный бэкап снапшотом) **до** `alembic upgrade` в проде. В этом репозитории pre-migrate бэкап делается в CI: шаг «Create backup before migration» в `.github/workflows/migrate.yml` (через SSH на хост) и/или `scripts/backup_before_migrate.sh` в пайплайне деплоя (см. `.github/workflows/deploy-environment.yml`).
- Храните дампы с понятным именем и TTL по политике команды; для инцидента сохраните путь к файлу в тикете.

### Восстановление (plain SQL от `pg_dump` без `-Fc`)

Остановите запись пользовательского трафика (maintenance / вывод из балансировщика), затем:

```bash
# пример: дамп на хосте, контейнер postgres как в compose
docker exec -i fittracker-postgres psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" < /path/to/pre_migration_YYYYMMDD_HHMMSS.sql
```

При необходимости **сначала** очистите/пересоздайте целевую БД по внутренней процедуре (осторожно: полное уничтожение текущих данных в БД).

Для формата **custom** (`pg_dump -Fc`) используйте **`pg_restore`** с флагами, согласованными с вашими ролями (`--no-owner`, `--if-exists` и т.д.).

После restore поднимите **тот же `IMAGE_TAG`**, который соответствует схеме в дампе, и выполните проверки из раздела ниже.

## Multi-step migration pattern (деструктивные и breaking changes)

Любое изменение, ломающее старый код или удаляющее данные, выносите в **несколько релизов**. Типичная схема из **трёх этапов**:

### Этап 1 — Additive (расширение схемы)

- Добавьте новые таблицы/колонки/индексы так, чтобы **старый** код продолжал работать (nullable, default, дублирование полей при переименовании).
- Код новой версии может начать писать в новые структуры, но не зависеть от удаления старых.

### Этап 2 — Deploy (переключение чтения/записи)

- Выкачайте версию приложения, которая использует новую схему и **перестаёт** опираться на устаревшие поля (или читает оба варианта во время двойной записи).
- Дождитесь стабильности: все инстансы на новой версии, метрики в норме.

### Этап 3 — Cleanup (сужение схемы)

- Отдельная миграция: удаление колонок/таблиц, жёсткие constraint’ы, удаление временных индексов.
- После этого этапа откат «назад кодом» без восстановления из бэкапа часто **невозможен** для пользовательских данных — снова опирайтесь на **pg_dump** и согласованный `IMAGE_TAG`.

Краткие правила совместимости: `docs/db/migration-safety.md`.

## Проверка после отката (smoke test)

Подставьте базовый URL API вместо `https://<API_HOST>` (и при необходимости порт). Ожидаются **HTTP 200** для успешной проверки готовности (кроме случаев, когда намеренно проверяете только liveness).

```bash
# Liveness (процесс жив)
curl -fsS "https://<API_HOST>/health/live"

# Корневой health (алиас логики сервиса)
curl -fsS "https://<API_HOST>/health"

# Readiness: БД + Redis (503 если degraded)
curl -fsS "https://<API_HOST>/health/ready"

# Версия v1 — те же пробы под префиксом API
curl -fsS "https://<API_HOST>/api/v1/system/live"
curl -fsS "https://<API_HOST>/api/v1/system/health"
curl -fsS "https://<API_HOST>/api/v1/system/ready"
curl -fsS "https://<API_HOST>/api/v1/system/version"
```

Для быстрой диагностики без `-f`: `curl -sS -o /dev/null -w "%{http_code}\n" "https://<API_HOST>/api/v1/system/ready"`.

Дополнительно: `docker compose -f docker-compose.prod.yml ps` на хосте — контейнеры в состоянии `healthy`.
