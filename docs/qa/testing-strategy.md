# Testing Strategy (QA) — FitTracker Pro

Документ определяет приоритеты тестирования, тестовую пирамиду проекта и список критичных сценариев (P0), которые должны быть покрыты автоматизацией и смок‑проверками.

## Цели

- Снизить риск регрессий в **критичных пользовательских потоках** (аутентификация Telegram, тренировки, здоровье, аналитика, офлайн‑синхронизация).
- Держать релиз‑готовность через быстрые проверки в CI и повторяемый смок в production.
- Максимум уверенности за минимальную стоимость: больше unit/интеграционных тестов, меньше «хрупких» UI E2E.

## Нефункциональные требования (качество)

- **Безопасность:** токены/сессии, CORS/Origin, секреты, Telegram `initData` валидация.
- **Надёжность:** идемпотентность критичных мутаций, корректное восстановление после сетевых ошибок.
- **Совместимость:** обратная совместимость API (в т.ч. период алиасов, см. `README.md`).
- **Производительность:** приемлемые времена ответа для основных экранов; отсутствие N+1 и «тяжёлых» запросов на главных потоках.

## Тестовая пирамида (целевое распределение)

Ориентировочные доли по числу тестов/прогонов (не догма; важнее покрытие рисков):

1. **Unit (60–70%)**
   - Backend: доменная логика, валидаторы, маппинг ошибок/исключений, вспомогательные функции, очереди/оркестрации без I/O.
   - Frontend: чистые функции, селекторы/утилиты, форматирование, маппинг DTO → view‑model, правила отображения конфиг‑режимов тренировок.

2. **Интеграционные (20–30%)**
   - Backend + Postgres/Redis: репозитории, транзакции, миграции, кеш/локи (если есть), ключевые application‑сценарии с реальными зависимостями.
   - Frontend ↔ API: «contract‑интеграции» на уровне клиента (моки/record‑replay допустимы), обработка ошибок и редиректов.

3. **Контрактные (5–10%)**
   - Проверки совместимости API схем (OpenAPI/DTO): обязательные поля, типы, backward‑compat изменения.
   - Особенно критично для `/api/v1/users/auth/*`, `/api/v1/workouts/*`, офлайн‑синхронизации.

4. **E2E/UI (3–8%)**
   - Минимальный набор P0 пользовательских путей: вход через Telegram, старт/завершение тренировки, базовая запись метрики, smoke основных страниц.
   - E2E тесты должны быть короткими, детерминированными и использовать стабильные селекторы.

5. **Ручное/исследовательское (по риску, регулярно)**
   - Telegram Mini App контекст, обновление PWA (prompt‑обновление), деградации офлайна, UX‑краевые случаи.

## Приоритеты покрытия

### P0 (блокер релиза)

- Любая поломка приводит к невозможности использования ключевых функций или к потере данных пользователя.
- Должно быть покрыто **автотестами** (unit+integration) и/или **E2E smoke**, а также входить в релизный смок.

### P1 (высокий приоритет)

- Ошибки заметны большинству пользователей, но есть обходной путь или ограниченный ущерб.
- Покрываем автотестами по мере появления регрессий/изменений, усиливаем контрактами.

### P2 (желательно)

- Редкие/косметические дефекты, локальные деградации, нестабильные сценарии.
- Покрываем выборочно (особенно если тест хрупкий/дорогой).

## Где запускаем тесты

- **CI (каждый PR):**
  - Frontend: `npm run lint`, `npm run test:ci`, `npm run build` (см. `docs/PRODUCTION_CHECKLIST.md`).
  - Backend: `pytest`.
- **Перед релизом/после деплоя (smoke):**
  - `GET /api/v1/system/health` и `GET /api/v1/system/version`.
  - Telegram login и `GET /api/v1/users/auth/me` (см. `docs/PRODUCTION_CHECKLIST.md`).

## Критичные сценарии (P0) — чеклист

Ниже список сценариев, которые считаются минимальным «щитком» от регрессий. Каждый сценарий должен иметь явный ожидаемый результат и критерии принятия.

### Аутентификация и сессия (Telegram)

- **Telegram login работает:**
  - `POST /api/v1/users/auth/telegram` принимает валидный Telegram `initData`, создаёт/возвращает токены.
- **Проверка текущего пользователя:**
  - `GET /api/v1/users/auth/me` возвращает профиль при валидном access token.
- **Невалидный/просроченный токен:**
  - корректные 401/403, клиент не попадает в бесконечные ретраи, UI показывает понятное состояние.

### Тренировки (core flow)

- **Список тренировок/сессий загружается** (после логина) без ошибок.
- **Старт тренировки**:
  - создание сессии/черновика (если предусмотрено), корректные начальные состояния таймера/отдыха.
- **Запись подходов/упражнений:**
  - добавление/изменение данных, корректные валидации (числа, отрицательные значения, обязательные поля).
- **Завершение тренировки:**
  - статус финализируется, данные сохраняются и отображаются в истории после перезагрузки.
- **Идемпотентность/повтор отправки:**
  - повторная отправка ключевой мутации (например, при таймауте/ретрае) не создаёт дублей и не ломает агрегаты.

### Каталог упражнений и справочники

- **Каталог упражнений доступен** (он часто нужен для тренировки).
- **Кеш/офлайн‑persist** не приводит к некорректным данным после обновления схемы (миграция ключей/очистка при несовместимости).

### Офлайн и синхронизация (PWA)

- **Офлайн‑режим после успешного онлайн визита:**
  - оболочка приложения открывается, основные экраны не «падают» из‑за отсутствия сети.
- **Очередь мутаций по тренировкам:**
  - действие в офлайне ставится в очередь и корректно синхронизируется при восстановлении сети.
- **Конфликт/повтор синка:**
  - повторный прогон очереди не приводит к порче данных (безопасная повторяемость).
- **Обновление PWA (registerType: prompt):**
  - новая версия не ломает текущую сессию; пользователь может обновиться вручную.

### Метрики здоровья (core health)

- **Создание/обновление базовой метрики** (через `/api/v1/health-metrics/*`):
  - валидации, сохранение, отображение на соответствующем экране.

### Аналитика (минимальный smoke)

- **Загрузка ключевой аналитики** (`/api/v1/analytics/*`) после логина:
  - корректная обработка пустых данных (новый пользователь) без 500 и без UI‑ошибок.

### Emergency (если включено пользователю)

- **Доступность раздела emergency**: базовая навигация/загрузка без падений.

### Платформа/эксплуатация

- **Health/version endpoints:**
  - `GET /api/v1/system/health` = 200 + ожидаемый JSON.
  - `GET /api/v1/system/version` = 200 + версия/sha присутствуют.
- **Миграции не ломают старт:**
  - после `alembic upgrade head` приложение поднимается и проходит smoke.

## Замечания по тест-дизайну (как избегать хрупкости)

- **Стабильные селекторы** для UI‑E2E (data-testid или эквивалент), минимум зависимости от текста/верстки.
- **Фиксировать договорённости API** контрактными тестами при любых изменениях DTO/эндпоинтов.
- **Ограничить E2E только P0**: всё остальное — через unit/интеграционные тесты и контрактные проверки.
